ML I/O benchmark · experiment 1 · May 2026

Warpdrive, I/O-heavy ML, and a local baseline

This is an effort to make Warpdrive blazing fast for I/O-heavy machine learning workloads, the kind where workloads spend more time moving data between systems than actually training. At LLM scale that shows up as multi-terabyte checkpoint saves, massive sharded datasets, and pipelines that must keep hardware continuously fed 1, 2. The same pressure appears outside generative models too: vision and recommender systems dealing with large shard caches, repeated epoch reads, and frequent artifact writes to a shared store 3.

The goal is to define a useful yet efficient system and not just build yet another blob store for the sake of showing that I've done it. Since my approach to building any system is product first you'll always see most engineering optimisations as an after thought.

What we run (and how it maps to production workloads)

Workload A (shard sweep) is the scenario in the next section: big opaque shards, concurrent workers, and repeated read passes over object storage or local files. In production that is closest to dataloader fleets hitting object storage (WebDataset-style shards, TFRecord or Parquet-backed training, or any pattern with concurrent GETs on multi-hundred-MB objects instead of one GET per tiny file).

Workload B (training-shaped): one serialized dataset blob, load it, train a small CNN on CPU, write checkpoints on a schedule, then stress fat checkpoint round-trips. In production this is closest to publishing a dataset or snapshot (large PUT), hydrating a node (large GET), and checkpointing model and optimizer state to shared storage for resume and collaboration. Training time still dominates the clock in our run, which is normal when the loop is non-trivial.

We compare Warpdrive to MinIO, each from a Docker image with an S3-compatible API over HTTP, as reproducible S3 peers. Both have deployment styles where they tend to look best, for example MinIO on fast local disks or a purpose-built distributed layout, and Warpdrive with its XFS-oriented local binary store and other production tuning when you are optimizing the full stack. This experiment does not chase those settings. We use one ordinary host layout, start the services in a straightforward way, and measure what the application sees: end-to-end wall time and client-observed latency for generic boto3 and PyTorch I/O. That is usually the signal teams care about before they redesign storage or networking.

We still run a local-disk benchmark alongside S3-compatible backends. I've picked local as a baseline for people who might ask why not keep everything in a local file system.

System details (experiment 1)
  • Hardware (benchmark host): Apple MacBook Air (Apple M3). CPU: 8 cores (4 performance, 4 efficiency). Memory: 16 GB unified. GPU: integrated Apple M3, 8 GPU cores, Metal 3.
  • OS: macOS 15.7 (build 24G419).
  • MinIO: Docker image; S3-compatible API over HTTP to a mapped port on loopback.
  • Warpdrive: Docker image; S3-compatible API over HTTP to a mapped port on loopback.
  • Local baseline: reads and writes on the host filesystem (no HTTP), for contrast with the S3 paths.
  • Client stack: boto3, PyTorch torch.save / torch.load, and the two workload scripts in this folder; chart numbers come from the JSON run reports saved alongside them.
  • Run conditions: single machine; Warpdrive, MinIO, and local paths exercised from the same Python process and wall clock.

Workload A: I/O-bound shard sweep

We use eight synthetic shards of 128 MiB each (about 1 GiB total), upload them once, then treat them like big training files on storage. Four workers run in parallel and each one downloads every shard in full for three rounds, which is about 3 GiB read and mimics a small fleet hammering object storage on large GETs. After each download the client checksums the bytes and runs a tiny PyTorch op so the loop still looks like ML, but the measured time is really GET throughput, not FLOPs. On local disk later rounds can get much faster once the OS cache warms; Warpdrive and MinIO still pay HTTP on every fetch, and that gap is what the charts show.

Each round reads all eight shards once (1 GiB); three rounds ⇒ 3 GiB. Round wall time is elapsed seconds for that full round (parallel workers). The MiB/s value is 3 GiB ÷ (sum of the three round times), an average for the whole read phase.

Workload B: Training-shaped I/O

65k synthetic 3×32×32 samples (roughly 770 MiB serialized as one dataset blob), 12 CPU epochs on a small CNN, checkpoint every 3 epochs, then three deliberate fat checkpoint save/load round-trips (~50 MiB class payloads) to stress PUT/GET of “big state.”

One phase where Warpdrive beat local and MinIO in our run is staging the single .pt blob. The script uses one S3 PutObject after torch.save. Warpdrive still handles the body as a chunked HTTP ingest, persists it with an append-oriented store and extent metadata.

Summary

On the training-oriented workload, Warpdrive outperformed both MinIO and local storage on dataset PUT time, and also surpassed MinIO during the large GET phase that repeatedly reloads the same blob. Although local storage achieved the lowest overall wall-clock time by bypassing the HTTP stack entirely, the large-blob dataset PUT phase was still faster on Warpdrive, showing that overall execution time alone does not capture the performance characteristics of individual workload phases.

The shard-sweep experiments further showed local storage scaling aggressively once the page cache was warmed. Between Warpdrive and MinIO, MinIO achieved slightly higher effective read throughput on this highly parallel access pattern in the evaluated session.

These experiments are part of a broader effort to identify the most effective interfaces for running ML workloads on object storage systems and to optimize read/write paths specifically for ML-driven access patterns rather than generic storage workloads.
